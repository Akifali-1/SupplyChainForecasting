✅ WEEK 7 — Basic Python Program on AWS Lambda
What you're doing: Write and run a simple Python function on AWS Lambda.
Steps:

Go to AWS Console → Services → Compute → Lambda
Click Create function
Select Author from scratch
Fill in:

Function name: basic-python-lambda
Runtime: Python 3.9
Architecture: x86_64


Under permissions → Select Use existing role → choose LabRole
Click Create function

Replace the default code with:
pythonimport json

def lambda_handler(event, context):
    a = event.get('num1', 0)
    b = event.get('num2', 0)
    result = a + b
    return {
        'statusCode': 200,
        'body': json.dumps({'sum': result})
    }

Click Deploy → wait for "Successfully deployed"
Click Test → Create new event

Event name: testInput
JSON body:



json   { "num1": 10, "num2": 20 }

Click Test → Output should show "sum": 30
Go to General Configuration → Edit → Set Memory: 128 MB, Timeout: 3 sec → Save


✅ WEEK 8 — SNS, SQS, and S3→SNS→SQS→Lambda
This week has 3 parts.

Part 1: SNS — Create Topic & Send Email

AWS Console → Search SNS → Open Amazon SNS
Left panel → Topics → Create topic

Type: Standard, Name: MyEmailTopic
Click Create topic


Open the topic → Subscriptions tab → Create subscription

Protocol: Email, Endpoint: your email address
Click Create subscription


Go to your email inbox → click Confirm subscription link
Back in SNS → Open topic → Click Publish message

Subject: Test Mail
Body: Hello this is SNS test
Click Publish message → You'll receive an email ✅




Part 2: SNS — Trigger Email on S3 Upload (S3 → SNS → Email)

Create S3 Bucket:

AWS Console → S3 → Create bucket
Name: my-upload-bucket-2026 (must be unique)
Region: same as your SNS topic
Click Create bucket


Reuse or create a new SNS topic (e.g., S3UploadNotification) with confirmed email subscription.
Add access policy to SNS topic (to allow S3 to publish):

Open SNS topic → Access policy → Edit
Paste the JSON below, replacing the 3 placeholders:



json   {
     "Version": "2012-10-17",
     "Statement": [{
       "Effect": "Allow",
       "Principal": { "Service": "s3.amazonaws.com" },
       "Action": "SNS:Publish",
       "Resource": "<SNS_TOPIC_ARN>",
       "Condition": {
         "ArnLike": { "aws:SourceArn": "<S3_BUCKET_ARN>" },
         "StringEquals": { "aws:SourceAccount": "<YOUR_ACCOUNT_ID>" }
       }
     }]
   }

Click Save


Configure S3 Event Notification:

Open your S3 bucket → Properties → scroll to Event notifications
Click Create event notification
Name: UploadNotification
Event types: ✅ All object create events
Destination: SNS topic → select your topic
Click Save changes


Upload any file to the bucket → you'll receive an email ✅


Part 3: S3 → SNS → SQS → Lambda (Full Pipeline)
Step 1: Create S3 Bucket

Name: mys3eventbucket123, same region throughout

Step 2: Create SNS Topic

Type: Standard, Name: MyS3SNSTopic
Copy the SNS Topic ARN

Step 3: Create SQS Queue

AWS Console → SQS → Create queue
Type: Standard, Name: MyS3Queue
Copy the Queue ARN

Step 4: Subscribe SQS to SNS

Open MyS3SNSTopic → Create subscription
Protocol: Amazon SQS, Endpoint: select MyS3Queue
Click Create subscription

Step 5: Add Access Policy to SQS (so SNS can send to it)

Open MyS3Queue → Access Policy → Edit
Replace SQS ARN and SNS ARN:

json{
  "Statement": [{
    "Effect": "Allow",
    "Principal": { "Service": "sns.amazonaws.com" },
    "Action": "sqs:SendMessage",
    "Resource": "<SQS_ARN>",
    "Condition": { "ArnEquals": { "aws:SourceArn": "<SNS_ARN>" } }
  }]
}
Step 6: Add Access Policy to SNS (same as Part 2 policy — allow S3 to publish)
Step 7: Configure S3 Event Notification

Open S3 bucket → Properties → Event notifications → Create event notification
Event type: All object create events
Destination: SNS → MyS3SNSTopic

Step 8: Test

Upload any file to the S3 bucket
Go to SQS → MyS3Queue → Send and receive messages → Poll for messages
You should see the S3 event notification message ✅

Step 9: Create Lambda Consumer

AWS Console → Lambda → Create Function
Name: SQSConsumerFunction, Runtime: Python 3.x
Click Create function
Paste this code:

pythondef lambda_handler(event, context):
    for record in event['Records']:
        print("Message received from SQS:")
        print(record['body'])
    return {'statusCode': 200, 'body': 'Message processed successfully'}

Click Deploy
Click Add Trigger → Select SQS → choose MyS3Queue → Add

Now uploading a file to S3 will automatically trigger Lambda via SNS → SQS ✅

✅ WEEK 9 — Elastic Load Balancer (ALB) + Auto Scaling Group

Part 1: Create Two EC2 Instances with Apache
Instance 1:

EC2 Console → Launch Instance

Name: webserver-1, AMI: Amazon Linux 2, Type: t2.micro
Security Group: Allow SSH (22) and HTTP (80)


Connect to instance via SSH and run:

bashyum update -y
yum install httpd -y
systemctl start httpd
systemctl enable httpd
echo "This is Server 1" > /var/www/html/index.html
Instance 2: Repeat, change name to webserver-2 and echo "This is Server 2".

Part 2: Create Application Load Balancer (ALB)

EC2 → Security Groups → Create Security Group

Name: lb-sg, Allow HTTP (80) from anywhere


EC2 → Load Balancers → Create Load Balancer → Application Load Balancer

Name: my-alb, Scheme: Internet-facing, IP type: IPv4
Listener: HTTP port 80
VPC: default, select all subnets
Security group: lb-sg


Create Target Group:

Name: web-servers-tg, type: Instance, Protocol: HTTP, Port: 80
Register targets: select webserver-1 and webserver-2 → Include as pending → Register


Click Create Load Balancer
Test: Copy the ALB DNS name → paste in browser → refresh multiple times → you'll see Server 1 and Server 2 alternating ✅


Part 3: Auto Scaling Group (ASG)

Create AMI from existing EC2:

Select webserver-1 → Actions → Image and Templates → Create Image
Name: my-webserver-AMI → wait until status: available


Create Launch Template:

EC2 → Launch Templates → Create Launch Template
Name: my-launch-template
AMI: my-webserver-AMI, Type: t2.micro
Security Group: lb-sg (HTTP 80)
Click Create


Create Auto Scaling Group:

EC2 → Auto Scaling Groups → Create Auto Scaling Group
Name: webserver-asg
Launch Template: my-launch-template
VPC: default, select all subnets
Attach to existing load balancer → select my-alb
Health check: HTTP, Grace period: 300 sec
Desired: 2, Min: 1, Max: 4
Click Create


Test Auto Scaling:

Go to EC2 Instances → manually Terminate one instance
Within ~1-2 min, ASG will automatically launch a replacement ✅

✅ WEEK 13 — IAM Roles (EC2 Access to S3 without Keys)
What you're doing: Attach a role to EC2 so it can access S3 without storing any credentials.

Create IAM Role:

AWS Console → IAM → Roles → Create role
Trusted entity: AWS Service
Use case: EC2 → Next
Search and select: AmazonS3FullAccess → Next
Name: EC2-S3-Role → Create role


Attach Role to EC2:

Go to EC2 Dashboard → select your running instance
Actions → Security → Modify IAM role
Select EC2-S3-Role → Update IAM role


Verify from inside the instance:

SSH into the instance
Run:



bash   aws s3 ls                    # ✅ WORKS — instance has S3 access via role
   aws ec2 describe-instances   # ❌ FAILS — no EC2 permission in role

No keys needed — EC2 gets temporary credentials automatically from the IAM Role ✅
